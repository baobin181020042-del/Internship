**Timeline:** 29/6 → 3/7 (5 working days)

> IRMS was completed by a five-member team. The notes below focus on my individual contribution while collaborating with the team.

## Day 1 - 29/6: IRMS Project Kick-off

**Work completed:** I attended the IRMS project kick-off meeting with my teammates to define the project objectives, scope, and core features of the Information Security Incident Management System. After discussing the project requirements, the team assigned responsibilities, and I was responsible for the backend development, including designing REST APIs, implementing data processing functions, and preparing the AWS development environment.

**Knowledge learned:** Clearly defining project scope and assigning responsibilities at the beginning helps ensure smooth collaboration and efficient development.

**Result achieved:** I understood my responsibilities within the team and successfully prepared the development environment for the project.

**Difficulty and lesson learned:** Effective communication among team members is essential to avoid overlapping responsibilities and maintain consistent API design.

## Day 2 - 30/6: REST API Design

**Work completed:** I analyzed the system requirements and designed RESTful APIs for the IRMS application. The APIs included Create, Read, Update, and Delete (CRUD) operations. I also worked with the frontend team to define a consistent request structure, response format, and HTTP status codes for seamless integration.

**Knowledge learned:** Designing APIs early in the development process reduces integration issues and minimizes future modifications.

**Result achieved:** I completed the REST API design and established a consistent data structure for communication between the backend and frontend.

**Difficulty and lesson learned:** Some business requirements needed further clarification before implementation to prevent frequent API changes.

## Day 3 - 1/7: Developing Incident CRUD APIs   

**Work completed:** I started implementing the Incident management APIs using the CRUD model, including creating, updating, retrieving, listing, and deleting incident records. I also added input validation to ensure that only valid data was processed and stored.

**Knowledge learned:** Input validation is essential for maintaining data integrity and preventing unexpected system errors.

**Result achieved:** I completed the basic CRUD APIs for incident management and verified their functionality using Postman.

**Difficulty and lesson learned:** Defining a standardized data model from the beginning makes future feature expansion much easier.

## Day 4 - 2/7: Timeline API and Error Handling

**Work completed:** I continued developing the Timeline API to record the activity history of each incident. In addition, I implemented validation and error-handling mechanisms to verify input data, handle exceptions, and return appropriate error messages for different scenarios.

**Knowledge learned:** Proper error handling improves system stability and helps both users and developers identify issues more efficiently.

**Result achieved:** I completed the Timeline APIs and established a consistent error-handling mechanism for the backend services.

**Difficulty and lesson learned:** Standardizing error responses early significantly reduces testing and maintenance effort.

## Day 5 - 3/7:  API Testing and Weekly Review

**Work completed:** I tested all implemented APIs using Postman, reviewed the backend source code, updated the API documentation, and coordinated with my teammates to prepare for frontend integration. I also documented the remaining issues and planned the tasks for the following week.

**Knowledge learned:** Continuous testing throughout development helps detect problems early and improves overall software quality.

**Result achieved:** The core backend APIs functioned correctly and were ready for integration with the frontend and other system components.

**Difficulty and lesson learned:** Regular communication between the backend and frontend teams is necessary to ensure a smooth integration process and minimize unexpected issues.
